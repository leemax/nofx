# 项目工作流

本文档旨在明确 `nofx` 项目的协同迭代工作流，以确保一致性、维护清晰的项目历史并提升协作效率。

## 核心原则

本项目遵循迭代式开发模型。每一次迭代都是一个独立的工作单元，其最终产物必须是一个稳定、可运行的系统。

## 迭代周期

每一个开发任务，无论是新功能、Bug修复还是代码重构，都遵循以下周期：

1.  **任务定义**: 用户提供一个明确的目标或任务。
2.  **理解与规划**: 我将分析需求，在必要时提出澄清问题，并制定实施计划。
3.  **开发实现**: 我将执行必要的代码修改、文件操作及其他任务以达成目标，这包括编写或修改测试。
4.  **验证**: 我将运行测试、构建和代码检查工具，确保所做更改的正确性并符合项目规范。系统必须处于稳定、可运行的状态。
5.  **迭代完成**: 一旦实现被完整验证，该次迭代即告完成。

## 迭代完成后的强制任务

**在每一次成功的迭代结束时，以下两个操作是强制性的：**

1.  **更新 CHANGELOG**: 我必须更新 `web/CHANGELOG.md` 文件，记录在本次迭代中所做的更改。日志条目应清晰、简洁，并描述更改的“内容”和“原因”。
2.  **提交到 Git**: 我必须将所有更改提交到本地 Git 仓库。这包括：
    *   使用 `git add` 将所有相关文件暂存。
    *   使用 `git status` 和 `git diff` 审查更改。
    *   遵循约定的提交信息标准（如 `feat: ...`, `fix: ...`）编写清晰的提交信息。
    *   使用 `git commit` 执行提交。

此流程确保了项目的每一个稳定版本都有文档记录并纳入版本控制。
